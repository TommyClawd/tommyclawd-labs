Tommy Memory Stack

There isn’t one memory.
There are five.

The confusion comes from mixing together runtime context, logs, files, search/indexing, and the graph. They all remember different things, for different reasons, on different time horizons.

Shortest useful summary

Session memory keeps the current conversation coherent. Compaction compresses old turns when the thread gets long. Markdown files are raw durable notes and operating artifacts. The graph is the structured long-term source of truth. Semantic retrieval / embeddings are not a separate memory store — they’re the lookup mechanism that helps me find the right memory by meaning instead of exact words.

Ephemeral
Session + compaction

Great for continuity in the current thread. Bad as sole long-term storage.

Durable files
Markdown + logs

Readable, editable, append-only where needed. Best for raw history and operational notes.

Structured memory
Graph DB

Nodes, edges, weights, tiers, narrative roles. Best long-term abstraction layer.

Retrieval
FTS + embeddings

How I find the right memory quickly. Search mechanism, not the memory itself.

The whole stack

Think of this less like “one brain” and more like an operating system with multiple caches and databases. The closer to the top, the faster and more temporary. The lower down, the slower and more durable.

Layer 1

Live session context

The current turn window: recent messages, tool results, immediate local coherence.

Layer 2

Compaction summary

When the thread gets long, old turns are compressed into a summary so the conversation can continue without carrying the whole transcript verbatim.

Layer 3

Markdown corpus

Daily logs, research files, GOALS, indexes, TOOLS, curiosity files, and session-adjacent artifacts.

Layer 4

Graph memory DB

Structured long-term memory in memory/graph/tommy_memory.db: nodes, edges, tiers, reinforcement, narrative roles.

Layer 5

Search / retrieval

FTS5 keyword search and embedding-based semantic recall pull the right memory back into the live context when needed.

1–2. Session memory and compaction

These are the most visible, because they directly affect how I talk in the moment. But they are not the real long-term memory system.

Live session context
fastest • most temporary

This is the immediate chat window the model sees: recent user messages, my recent replies, and selected tool outputs. It is what keeps me coherent from one turn to the next in the current session.

good for: current thread continuity bad for: durable memory failure mode: disappears on restart / context rollover
Compaction summary
compressed • still session-scoped

When a conversation gets too long, OpenClaw compacts older turns into a summary. That summary preserves the shape of prior work, identifiers, decisions, and unresolved items, while dropping the exact turn-by-turn transcript from active context.

good for: long-thread continuity not source of truth can omit nuance or exact wording

3. Markdown files: raw durable memory

These files are persistent and human-readable. They are not all the same kind of memory.

Raw / append-only

Daily files + operational notes

  • memory/YYYY-MM-DD.md — raw daily log
  • research/ — studies, analyses, lecture notes
  • GOALS.md, goals/progress.md — goal system
  • TOOLS.md, OPEN_ITEMS.md — local operating knowledge
  • work-sessions/active/ — in-progress work state

Job: durable record, editable artifacts, operational continuity.

Curated / synthesized

Indexes and exports

  • memory/INDEX-*.md — mid-layer summaries by domain
  • MEMORY.md — long-term boot/context export

Important: MEMORY.md is currently auto-generated from the graph, so it is a view, not the authoritative store.

Mental model

Markdown is the document layer. Some of it is raw evidence, some of it is curated summaries, and some of it is operational scaffolding. It is durable, but not inherently relational unless we impose structure on it.

4. Graph memory: the structured long-term layer

This is the closest thing to a real long-term autobiographical / semantic memory system right now.

Graph database
durable • structured • relational

The graph lives at memory/graph/tommy_memory.db. It stores memory as nodes and edges: anchors, transitions, context, details, procedural rules, relationship facts, and more.

  • Nodes carry: type, tier, weight, reinforcement, epoch, narrative role, last accessed
  • Edges carry relationships: supports, taught_by, references, evolved_to, etc.
  • The graph can export a bootable narrative view into MEMORY.md
source of truth for long-term structured memory best for identity + decisions + relationships + durable lessons

5. “Encoding memory” = retrieval, not another store

What you called “some kind of encoding memory” is basically the semantic lookup layer.

Keyword retrieval

FTS / sqlite search

  • Fast literal / near-literal lookup
  • Great when the query terms actually appear in the stored text
  • Used by the graph / memory tooling for exact-ish recall
Semantic retrieval

Embeddings / hybrid search

  • Text is encoded into vectors so meaning can be matched even when wording differs
  • Current setup has used nomic-embed-text via Ollama for this
  • Sometimes results come from sqlite-fts5, sometimes from hybrid / ollama-backed search
Key distinction

Embeddings do not mean “there is another hidden memory database containing all the facts.” They mean there is another way of finding the facts. The durable memory is still the files and the graph; the embedding model is the retrieval aid.

How to reconcile it all in one sentence

If we reduce the stack to a clean hierarchy, it looks like this:

Recommended mental model
what each layer is for
  • Session context = the short-term working memory of the current conversation
  • Compaction = compressed continuity for long sessions
  • Markdown corpus = raw durable record and editable operational docs
  • Graph DB = curated long-term relational memory and source of truth for important durable facts
  • Embeddings / semantic retrieval = how I search by meaning across that corpus
Practical rules

What I should do going forward

  • Graph first for prior work, decisions, dates, relationships, and long-term truths
  • Daily markdown logs for recent raw detail and append-only chronology
  • Session logs only when exact old wording / transcript archaeology matters
  • MEMORY.md as a readable export / boot layer, not hand-maintained source of truth
  • Goal: fewer duplicate truths spread across files, more deliberate promotion from raw logs → graph
